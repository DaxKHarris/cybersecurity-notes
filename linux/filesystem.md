# Filesystem

## Structure

The Linux filesystem is hierarchical — a tree starting at root (`/`). Everything branches from there, including other drives and devices which are mounted as directories rather than separate drive letters.

## Key Directories

| Directory | Purpose |
|-----------|---------|
| `/etc` | Configuration files for the system and installed software |
| `/var` | Variable data — logs, databases, mail, spool files |
| `/var/log` | System and service logs |
| `/home` | User home directories (`/home/username`) |
| `/root` | Home directory for the root user |
| `/bin` | Essential user binaries (ls, cp, mv, etc.) |
| `/sbin` | System binaries, typically for root (fdisk, iptables, etc.) |
| `/usr` | Secondary hierarchy — most installed software lives here |
| `/usr/bin` | Non-essential user binaries |
| `/lib` / `/lib64` | Shared libraries required by binaries |
| `/boot` | Boot loader files, kernel images |
| `/dev` | Device files — hardware is represented as files here |
| `/proc` | Virtual filesystem — exposes running process and kernel info |
| `/sys` | Virtual filesystem — exposes kernel and hardware information |
| `/opt` | Optional/third-party software not managed by the package manager |
| `/tmp` | Temporary files, world-writable, cleared on reboot |
| `/mnt` | Temporary mount point for manually mounted drives |
| `/media` | Mount point for removable media (USB drives, etc.) |

## Security Relevant Directories

### /tmp
World-writable — any user can write and execute files here. A common attacker technique is dropping malicious files in `/tmp` because it is always writable regardless of privilege level. Monitor for unexpected executables here.

### /proc
Not real files — generated by the kernel on the fly. `/proc/[pid]/` exposes everything about a running process: open files, network connections, memory maps. Useful for forensics on a potentially compromised system.

### /dev
Hardware is represented as files. `/dev/sda` is a hard drive, `/dev/nvme0n1` is an NVMe drive, `/dev/null` discards anything written to it. Not driver files — drivers live in the kernel.

### /etc/systemd/system and /lib/systemd/system
Where systemd service files live. A common attacker persistence technique is dropping a malicious `.service` file here to survive reboots. Know where these are and audit them if a system is suspected compromised.
